Jetpack Compose UI: Leading a Design System from Scratch

Wine Chit Paing · March 2026

When I first proposed building a complete design system with Jetpack Compose across multiple production apps at Codigo, the response was a mix of excitement and skepticism. Compose was still maturing, and betting our entire UI layer on it was a bold move. Two years later, that decision became one of the most impactful technical choices I've made in my career.

Why Build a Design System?

At Codigo, we were maintaining six Android apps simultaneously — Kcuts Go, Boulder Movement, Yoga Movement, SPOTV NOW, AIA+, and Rezeve. Each had its own styling, custom views, and XML layouts. Every design change meant updating six codebases. Bug fixes in one app's UI didn't automatically carry over to others.

The inconsistency was costing us time, introducing bugs, and making onboarding new developers painfully slow. We needed a shared language — a design system that could unify our apps while still allowing each to have its own identity.

Starting with Foundations

I began with the atomic design methodology, adapted for Compose:

  • Tokens: Colors, typography, spacing, and elevation defined as Compose theme values. Each app could override these tokens to maintain its brand identity.
  • Atoms: Basic composables — buttons, text fields, chips, icons. These were the smallest reusable units.
  • Molecules: Combinations of atoms — search bars, list items, card headers.
  • Organisms: Complex UI sections — booking cards, payment forms, video player controls.

The Architecture Behind It

The design system lived in a shared Kotlin module that each app included as a dependency:

// design-system module structure
designsystem/
├── theme/
│   ├── AppTheme.kt
│   ├── Color.kt
│   ├── Typography.kt
│   └── Shape.kt
├── components/
│   ├── buttons/
│   ├── inputs/
│   ├── cards/
│   └── navigation/
└── tokens/
    ├── Spacing.kt
    └── Elevation.kt

Each app would wrap the shared theme with its own brand tokens:

@Composable
fun KcutsGoTheme(content: @Composable () -> Unit) {
    AppTheme(
        colors = kcutsColorScheme,
        typography = kcutsTypography,
        content = content
    )
}

Challenges We Faced

1. Compose Was Still Evolving

APIs changed between alpha, beta, and stable releases. We had to maintain a migration guide and dedicate time each sprint to updating deprecated APIs. The key lesson: pin your Compose BOM version and upgrade deliberately, not reactively.

2. Convincing the Team

Not everyone was comfortable with Compose initially. I organized weekly "Compose labs" — pair programming sessions where we'd rebuild an existing XML screen in Compose together. Seeing a 200-line XML layout reduced to 60 lines of declarative code was usually enough to convert skeptics.

3. Performance Pitfalls

Early on, we hit recomposition issues. Lists would lag, animations would stutter. I established strict rules:

  • Always use remember and derivedStateOf for computed values
  • Mark stable classes with @Immutable or @Stable
  • Use key() in LazyColumn items
  • Profile with Layout Inspector before and after changes

The Results

After fully migrating to the Compose design system:

  • Development speed increased by ~40% for new features — components were pre-built and tested.
  • UI bugs dropped by 60% — shared components meant fixes applied everywhere.
  • Onboarding time halved — new developers learned one system, not six different UI approaches.
  • All apps maintained 98%+ crash-free rates throughout the migration.

What I'd Do Differently

If I were starting today, I'd invest more in Compose Previews from day one. We added preview annotations late, and it slowed our design review process. I'd also adopt Compose Multiplatform earlier — the shared UI layer we built for Android would have been a natural fit for iOS, which is exactly what I'm doing now at Gradion Thailand.

A design system isn't just a component library. It's a shared vocabulary between designers and developers — a contract that says "this is how we build things here."

Leading this initiative taught me that the hardest part of a design system isn't the code — it's the alignment. Getting designers, developers, and product managers to agree on constraints and conventions requires patience, communication, and a willingness to iterate. The technical implementation is the easy part.
