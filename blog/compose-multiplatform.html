Going Multiplatform: Rewriting an App with Compose Multiplatform

Wine Chit Paing · September 2025

When the SBS 2026 app (formerly NFQ Summit) needed to support both Android and iOS, we had a choice: maintain two native codebases, adopt Flutter, or try something newer — Compose Multiplatform. We chose the last option, and it fundamentally changed how I think about cross-platform development.

Why Compose Multiplatform Over Flutter?

The decision wasn't obvious. Flutter has a larger ecosystem and more battle-tested production apps. But our team had two strong advantages with Compose Multiplatform:

  • Existing Kotlin expertise: Our Android team was already fluent in Kotlin and Jetpack Compose. Zero learning curve for the UI layer.
  • Shared business logic: Kotlin Multiplatform (KMP) let us share not just UI, but networking, data models, and business logic. Flutter would have required rewriting everything in Dart.
  • Native interop: For platform-specific features like NFC, we could drop into native code seamlessly using expect/actual declarations.

The Migration Strategy

We didn't rewrite everything at once. The migration happened in three phases:

Phase 1: Shared Data Layer (2 weeks)

We extracted models, API services, and repositories into a shared KMP module. This was the lowest-risk, highest-value move:

// shared/src/commonMain
data class SummitEvent(
    val id: String,
    val title: String,
    val speaker: String,
    val startTime: Instant,
    val room: String
)

// shared/src/commonMain
class EventRepository(
    private val apiClient: ApiClient
) {
    suspend fun getEvents(): List<SummitEvent> {
        return apiClient.get("/events").body()
    }
}

Phase 2: Shared UI Components (3 weeks)

We moved Compose UI components into the shared module. Most screens worked identically on both platforms. The project structure looked like:

shared/
├── src/
│   ├── commonMain/
│   │   ├── models/
│   │   ├── network/
│   │   └── ui/
│   │       ├── components/
│   │       ├── screens/
│   │       └── theme/
│   ├── androidMain/    # Platform-specific implementations
│   └── iosMain/        # Platform-specific implementations
androidApp/             # Android entry point
iosApp/                 # iOS entry point (SwiftUI wrapper)

Phase 3: Platform-Specific Features (2 weeks)

NFC functionality, push notifications, and deep linking required platform-specific code. The expect/actual pattern made this clean:

// commonMain
expect class NfcReader {
    fun startScanning()
    fun stopScanning()
    val scannedTag: StateFlow<NfcTag?>
}

// androidMain
actual class NfcReader(context: Context) {
    // Android NFC implementation
}

// iosMain
actual class NfcReader() {
    // CoreNFC implementation
}

What Worked Brilliantly

UI Code Sharing: ~85%

The vast majority of our UI code is shared. Screens, components, themes, navigation — all written once, running on both platforms. The Compose rendering engine produces truly native-feeling results on iOS, which surprised me.

Single Source of Truth for Business Logic

No more "did we fix that bug on iOS too?" Every fix to shared code applies everywhere. This alone justified the migration.

Developer Velocity

Our team of Android developers could now ship iOS features without learning Swift from scratch. The iOS-specific code (AppDelegate, SwiftUI wrapper) was minimal — maybe 500 lines total.

What Was Challenging

iOS-Specific UI Expectations

iOS users expect certain platform conventions — swipe-to-go-back, specific scroll physics, SF Symbols. We had to add platform checks for these:

@Composable
fun PlatformBackHandler(onBack: () -> Unit) {
    if (getPlatform() == Platform.IOS) {
        // iOS handles swipe-to-back natively
    } else {
        BackHandler(onBack = onBack)
    }
}

Build Times

KMP builds are slower than single-platform builds, especially for iOS. We mitigated this with build caching and by running iOS builds less frequently during development — most UI work could be previewed on Android first.

Library Ecosystem

Not every Android library has a KMP equivalent. We had to find or build multiplatform alternatives for image loading, local storage, and analytics. The ecosystem is growing fast, but gaps still exist.

Results

  • 85% code shared between Android and iOS
  • Development time reduced by ~50% compared to maintaining two native codebases
  • Single team delivering both platforms instead of separate Android and iOS teams
  • Production-grade app deployed to both App Store and Play Store

Would I Do It Again?

Absolutely. Compose Multiplatform is now my recommended approach for teams that already know Kotlin. For the Zsolt Lite project at Gradion Thailand, we chose Flutter instead because it was a client requirement and the team wanted to expand their Flutter expertise. Both are valid choices — the right tool depends on the team and context.

Cross-platform doesn't mean compromise. With Compose Multiplatform, we shipped a better app faster — not a watered-down version of two native apps.

The future of mobile development is multiplatform. The question isn't whether to share code between platforms — it's how much and with which tool. For Kotlin teams, Compose Multiplatform is the answer I've been waiting for.
