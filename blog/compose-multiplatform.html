Going Multiplatform: Rewriting an App with Compose Multiplatform

Wine Chit Paing · March 2026

When the SBS 2026 app (formerly NFQ Summit) needed to support both Android and iOS, I took full ownership of the rewrite — architecture, implementation, and delivery. No separate iOS team, no pair programming partner — just me, Kotlin 2.2, Claude Code, and a deadline. I chose Compose Multiplatform as the framework and Claude Code as my AI-powered development partner. Together, this combination fundamentally changed how I think about cross-platform development.

Why Compose Multiplatform Over Flutter?

The decision wasn't obvious. Flutter has a larger ecosystem and more battle-tested production apps. But as a solo developer on this project, I had clear advantages with Compose Multiplatform:

  • My existing Kotlin expertise: I was already fluent in Kotlin and Jetpack Compose from years of Android development. Zero learning curve for the UI layer meant I could move fast.
  • Shared business logic: KMP let me share not just UI, but networking (Ktor), data models (Kotlinx Serialization), database (Room), and DI (Koin). Flutter would have required rewriting everything in Dart — a luxury I didn't have as one developer.
  • Native interop: For platform-specific features like QR scanning and alarm scheduling, I could drop into native code seamlessly using expect/actual declarations.

AI-Powered Development with Claude Code

Being a solo developer on a cross-platform rewrite sounds impossible — until you add AI to the equation. I used Claude Code throughout the entire project, orchestrating multiple specialized agents in parallel:

  • Architecture planning: I'd describe a feature requirement and have Claude Code help me design the KMP module structure, expect/actual declarations, and shared composable hierarchy before writing a single line.
  • Code generation: Boilerplate-heavy tasks like creating Room entities, DAOs, Ktor API services, and Koin modules across commonMain, androidMain, and iosMain were dramatically accelerated.
  • Platform-specific code: Claude Code helped me implement the Android QR scanner with ML Kit and ZXing fallback, and the iOS scanner with AVFoundation — two completely different implementations sharing one interface.
  • Code review: Every major feature went through Claude Code review — catching recomposition issues, lifecycle management bugs, and platform-specific gotchas before they shipped.
  • ktlint compliance: Claude Code ensured all generated code passed ktlintCheck with the project's strict formatting rules (120-char line length, Compose naming conventions).

Claude Code didn't replace my decision-making — it amplified my output. I still made every architectural choice, but the time from decision to implementation was cut dramatically.

Project Structure

The app follows Clean Architecture with a clear separation between data, domain, and presentation:

composeApp/
├── src/
│   ├── commonMain/kotlin/com/nfq/nfqsummit/
│   │   ├── data/           # Room entities, DAOs, data sources
│   │   ├── domain/         # Business logic and models
│   │   ├── presentation/   # Screens, components, navigation
│   │   └── utils/          # Utility functions and helpers
│   ├── androidMain/        # Android-specific implementations
│   └── iosMain/            # iOS-specific implementations
iosApp/                     # iOS entry point (SwiftUI wrapper)

The Hard Parts: Platform-Specific Features

QR Code Scanner — Two Platforms, Two Implementations

This was the most complex platform-specific feature. On Android, I needed two different implementations depending on API level:

  • Android 12+ (API 31+): Google ML Kit with Google Code Scanner — the modern, reliable approach
  • Older Android: Custom ZXing-based scanner supporting QR, Code128, EAN, UPC, and more
  • iOS: Native AVFoundation camera with custom overlay UI, permission dialogs, and settings integration

Claude Code was invaluable here — it helped me implement the AVFoundation scanner on iOS without needing deep UIKit/SwiftUI expertise, while I focused on the Android ML Kit integration I knew well.

What Worked Brilliantly

UI Code Sharing: ~95%

Dashboard, Schedule, Explore, Blog, QR Scanner screens — all shared Compose code. Material3 theming with custom Gradion branding worked identically on both platforms. Navigation Compose handled routing seamlessly. The Compose rendering engine delivers smooth, native-quality performance on iOS — users can't tell it wasn't built with SwiftUI.

Solo Developer + AI Velocity

Being a single developer delivering both platforms sounds daunting, but Compose Multiplatform combined with Claude Code made it not just manageable — it was fast. I could describe a feature, have Claude Code scaffold the shared and platform-specific layers, then focus my time on refining the UX and business logic. I went from being "the Android guy" to shipping a full cross-platform production app single-handedly, with AI as my tireless pair programmer.

What Was Challenging

iOS-Specific UI Expectations

iOS users expect certain platform conventions — swipe-to-go-back, specific scroll physics, native permission dialogs with settings redirect. I had to add platform checks and custom SwiftUI integration for these edge cases.

Build Times

KMP builds are slower than single-platform builds, especially for iOS. I worked around this with Gradle build caching and by running iOS builds less frequently during development — most UI work could be previewed on Android first.

Library Compatibility

Not every Android library has a KMP equivalent. I chose carefully: Coil3 for images (multiplatform-ready), Ktor for networking (platform engines), Moko for permissions. For QR scanning, I accepted the need for fully separate implementations — some things are just inherently platform-specific.

Results

  • 95% code shared between Android and iOS
  • 5 major screens — Dashboard, Schedule, Explore, Blog, QR Scanner
  • Full offline support with Room database
  • One developer + Claude Code delivering both platforms — no separate teams needed
  • Production-grade app deployed to both App Store and Play Store
  • Android API 24+ and iOS 13.0+ support

Would I Do It Again?

Without hesitation. Compose Multiplatform with Kotlin 2.2, Room, Ktor, and Koin gave me a production-ready stack that shares almost everything. Claude Code turned what would have been a 3-person, 3-month project into a solo effort delivered on schedule.

For the Zsolt Lite project at Gradion Thailand, I chose Flutter instead because it was a client requirement and I wanted to expand my Flutter expertise. Both are valid choices — the right tool depends on the context.

Cross-platform doesn't mean compromise. With Compose Multiplatform and Claude Code, I shipped a better app faster as a solo developer — not a watered-down version of two native apps.

The future of mobile development is multiplatform — and AI-assisted. For Kotlin developers, Compose Multiplatform is the framework answer. And for solo developers who need to punch above their weight, Claude Code is the force multiplier that makes ambitious projects possible.
