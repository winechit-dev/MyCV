Going Multiplatform: Rewriting an App with Compose Multiplatform

Wine Chit Paing · March 2026

When the SBS 2026 app (formerly NFQ Summit) needed to support both Android and iOS, I took full ownership of the rewrite — architecture, implementation, and delivery. No separate iOS team, no pair programming partner — just me, Kotlin, Claude Code, and a deadline. I chose Compose Multiplatform as the framework and Claude Code as my AI-powered development partner. Together, this combination fundamentally changed how I think about cross-platform development.

Why Compose Multiplatform Over Flutter?

The decision wasn't obvious. Flutter has a larger ecosystem and more battle-tested production apps. But as a solo developer on this project, I had clear advantages with Compose Multiplatform:

  • My existing Kotlin expertise: I was already fluent in Kotlin and Jetpack Compose from years of Android development. Zero learning curve for the UI layer meant I could move fast.
  • Shared business logic: Kotlin Multiplatform (KMP) let me share not just UI, but networking, data models, and business logic. Flutter would have required rewriting everything in Dart — a luxury I didn't have as one developer.
  • Native interop: For platform-specific features like alarm scheduling, I could drop into native code seamlessly using expect/actual declarations.

The Secret Weapon: Claude Code

Being a solo developer on a cross-platform rewrite sounds impossible — until you add AI to the equation. I used Claude Code throughout the entire project, orchestrating multiple specialized agents in parallel:

  • Architecture planning: I'd describe a feature requirement and have Claude Code help me design the KMP module structure, expect/actual declarations, and shared composable hierarchy before writing a single line.
  • Code generation: Boilerplate-heavy tasks like creating data models, API services, and repository layers across commonMain, androidMain, and iosMain were dramatically accelerated.
  • Code review: Every major feature went through Claude Code review before integration — catching potential issues with recomposition, lifecycle management, and platform-specific gotchas.
  • Debugging: When iOS-specific rendering issues appeared, I'd describe the behavior and get targeted solutions without needing deep iOS/UIKit expertise.

Claude Code didn't replace my decision-making — it amplified my output. I still made every architectural choice, but the time from decision to implementation was cut dramatically.

The Migration Strategy

I didn't rewrite everything at once. Working solo with Claude Code as my AI assistant, I planned the migration in three focused phases to manage risk:

Phase 1: Shared Data Layer (2 weeks)

I extracted models, API services, and repositories into a shared KMP module. This was the lowest-risk, highest-value move:

// shared/src/commonMain
data class SummitEvent(
    val id: String,
    val title: String,
    val speaker: String,
    val startTime: Instant,
    val room: String
)

// shared/src/commonMain
class EventRepository(
    private val apiClient: ApiClient
) {
    suspend fun getEvents(): List<SummitEvent> {
        return apiClient.get("/events").body()
    }
}

Phase 2: Shared UI Components (3 weeks)

I moved Compose UI components into the shared module. Most screens worked identically on both platforms. The project structure looked like:

shared/
├── src/
│   ├── commonMain/
│   │   ├── models/
│   │   ├── network/
│   │   └── ui/
│   │       ├── components/
│   │       ├── screens/
│   │       └── theme/
│   ├── androidMain/    # Platform-specific implementations
│   └── iosMain/        # Platform-specific implementations
androidApp/             # Android entry point
iosApp/                 # iOS entry point (SwiftUI wrapper)

Phase 3: Platform-Specific Features (2 weeks)

Alarm scheduling, deep linking, and platform permissions required platform-specific code. The expect/actual pattern made this clean:

// commonMain
expect class AlarmScheduler {
    fun schedule(title: String, body: String, triggerAt: Instant)
    fun cancel(alarmId: String)
    val scheduledAlarms: StateFlow<List<Alarm>>
}

// androidMain
actual class AlarmScheduler(context: Context) {
    // AlarmManager + BroadcastReceiver implementation
}

// iosMain
actual class AlarmScheduler() {
    // UNUserNotificationCenter + UNCalendarNotificationTrigger implementation
}

What Worked Brilliantly

UI Code Sharing: ~95%

The vast majority of the UI code is shared. Screens, components, themes, navigation — all written once, running on both platforms. The Compose rendering engine produces truly native-feeling results on iOS, which surprised me.

Single Source of Truth for Business Logic

No more worrying about "did I fix that bug on iOS too?" Every fix to shared code applies everywhere. As a solo developer, this was the biggest win — I didn't have to maintain two separate codebases with diverging logic.

Solo Developer + AI Velocity

Being a single developer delivering both platforms sounds daunting, but Compose Multiplatform combined with Claude Code made it not just manageable — it was fast. I could describe a feature, have Claude Code scaffold the shared and platform-specific layers, then focus my time on refining the UX and business logic. The iOS-specific code (AppDelegate, SwiftUI wrapper) was minimal — maybe 500 lines total. I went from being "the Android guy" to shipping a full cross-platform app single-handedly, with AI as my tireless pair programmer.

What Was Challenging

iOS-Specific UI Expectations

iOS users expect certain platform conventions — swipe-to-go-back, specific scroll physics, SF Symbols. I had to add platform checks for these:

@Composable
fun PlatformBackHandler(onBack: () -> Unit) {
    if (getPlatform() == Platform.IOS) {
        // iOS handles swipe-to-back natively
    } else {
        BackHandler(onBack = onBack)
    }
}

Build Times

KMP builds are slower than single-platform builds, especially for iOS. I mitigated this with build caching and by running iOS builds less frequently during development — most UI work could be previewed on Android first.

Library Ecosystem

Not every Android library has a KMP equivalent. I had to find or build multiplatform alternatives for image loading, local storage, and analytics. The ecosystem is growing fast, but gaps still exist.

Results

  • 95% code shared between Android and iOS
  • Development time reduced by ~50% compared to maintaining two native codebases
  • One developer + Claude Code delivering both platforms — no separate Android and iOS teams needed
  • Production-grade app deployed to both App Store and Play Store

Would I Do It Again?

Absolutely. Compose Multiplatform is now my recommended approach for developers who already know Kotlin. For the Zsolt Lite project at Gradion Thailand, I chose Flutter instead because it was a client requirement and I wanted to expand my Flutter expertise. Both are valid choices — the right tool depends on the context.

Cross-platform doesn't mean compromise. With Compose Multiplatform and Claude Code, I shipped a better app faster as a solo developer — not a watered-down version of two native apps.

The future of mobile development is multiplatform — and AI-assisted. The question isn't whether to share code between platforms — it's how much and with which tools. For Kotlin developers, Compose Multiplatform is the framework answer. And for solo developers who need to punch above their weight, Claude Code is the force multiplier that makes ambitious projects possible.
