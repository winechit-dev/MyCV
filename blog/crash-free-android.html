Achieving 98% Crash-Free: My Approach to Android Stability

Wine Chit Paing · November 2025

Across multiple production apps — Kcuts Go (54K+ MAU), Boulder Movement (5.8K+ MAU), and Yoga Movement (14K+ MAU) — I've consistently maintained crash-free rates above 98%. This isn't luck. It's a systematic approach to stability that I've refined over six years of shipping Android apps.

The Foundation: Crash Prevention Starts at Architecture

Most crashes I've debugged in my career weren't caused by obscure edge cases. They were caused by predictable problems: null pointers, unhandled exceptions in network calls, lifecycle mismanagement. Good architecture eliminates entire categories of crashes before they happen.

Kotlin-First, Null-Safe by Default

Migrating from Java to Kotlin was the single biggest crash reduction move. Kotlin's type system catches null-related issues at compile time. In the Mother Finance app (my last Java-heavy project), ~35% of crashes were NullPointerException. In Kcuts Go (pure Kotlin), NPEs are virtually zero.

Structured Concurrency with Coroutines

Every async operation is scoped to the appropriate lifecycle:

class BookingViewModel(
    private val repository: BookingRepository
) : ViewModel() {

    fun confirmBooking(slot: TimeSlot) {
        viewModelScope.launch {
            try {
                val result = repository.confirm(slot)
                _uiState.update { it.copy(confirmation = result) }
            } catch (e: CancellationException) {
                throw e // Never swallow cancellation
            } catch (e: Exception) {
                _uiState.update { it.copy(error = e.toUserMessage()) }
                crashlytics.recordException(e)
            }
        }
    }
}

Key rules: never use GlobalScope, always let CancellationException propagate, and scope to viewModelScope or lifecycleScope.

Testing Strategy: The Three Layers

Layer 1: Unit Tests with JUnit 4

Every ViewModel and Repository gets unit tests. I focus on edge cases that would cause crashes in production:

  • Empty lists and null responses from APIs
  • Network timeouts and malformed JSON
  • State transitions under rapid user interaction
  • Concurrent operations on shared state

Layer 2: Integration Tests

I test the full data flow from API to UI state. Using MockWebServer, I simulate real server responses — including error responses, slow responses, and partial data. This catches serialization issues and error handling gaps that unit tests miss.

Layer 3: UI Tests with Espresso

Critical user flows get Espresso tests. For Kcuts Go, that means:

  • Complete booking flow: browse → select slot → pay → confirm
  • Error recovery: network failure during payment → retry → success
  • Edge cases: back navigation during loading, rapid button taps

Proactive Crash Monitoring

Firebase Crashlytics: Beyond Default Setup

I don't just install Crashlytics — I instrument it:

// Custom keys for every screen
crashlytics.setCustomKey("screen", "BookingConfirmation")
crashlytics.setCustomKey("slot_id", slot.id)
crashlytics.setCustomKey("payment_method", method.name)

// Non-fatal logging for suspicious states
if (slots.isEmpty() && !isFirstLoad) {
    crashlytics.recordException(
        IllegalStateException("Empty slots on reload - possible API issue")
    )
}

Custom keys transform a generic stack trace into a complete debugging context. When a crash happens, I know exactly which screen, which data, and which user action triggered it.

The "Zero New Crashes" Sprint Rule

At Codigo, I introduced a rule: no sprint ships with known new crashes. If Crashlytics shows a new crash cluster after a release, it gets hotfixed before the next feature work begins. This sounds strict, but it keeps the crash-free rate from slowly degrading — the most common failure mode I've seen in Android teams.

Common Crash Categories and Fixes

1. Fragment/Activity Lifecycle Crashes

The classic IllegalStateException: Can not perform this action after onSaveInstanceState. Solution: use Lifecycle-aware components and never commit fragments from background callbacks.

2. RecyclerView / LazyColumn IndexOutOfBounds

Usually caused by list mutations during scroll. Solution: immutable lists with DiffUtil (RecyclerView) or proper key() usage (Compose).

3. OOM on Image Loading

Loading full-resolution images into ImageViews without resizing. Solution: use Coil or Glide with proper sizing, and add memory monitoring in debug builds.

4. SSL/Network Crashes

Unhandled SSLHandshakeException or SocketTimeoutException. Solution: wrap all network calls with proper error handling and configure sensible timeouts in OkHttp.

The Mindset Shift

Early in my career, I treated stability as something you fix after launch. Now I treat it as something you design for before writing the first line of code. Every architectural decision, every API integration, every state management choice is evaluated through the lens of "how will this fail?"

A 98% crash-free rate doesn't mean your app is perfect. It means you've built systems that catch the remaining 2% before your users do.

The apps I'm most proud of aren't the ones with the most features — they're the ones where users never think about stability because it just works.
