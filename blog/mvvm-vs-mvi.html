MVVM vs MVI: Choosing the Right Architecture for Your Android App

Wine Chit Paing · January 2023

After building production Android apps with both MVVM and MVI for over six years, I've developed strong opinions about when to use each. This isn't a theoretical comparison — it's a practical guide drawn from real apps serving tens of thousands of users.

MVVM: The Workhorse

Model-View-ViewModel has been my default architecture for most projects. It's straightforward, well-documented, and aligns perfectly with Android's lifecycle-aware components.

How I Structure MVVM

class BookingViewModel(
    private val repository: BookingRepository
) : ViewModel() {

    private val _uiState = MutableStateFlow(BookingUiState())
    val uiState: StateFlow<BookingUiState> = _uiState.asStateFlow()

    fun loadAvailableSlots(date: LocalDate) {
        viewModelScope.launch {
            _uiState.update { it.copy(isLoading = true) }
            repository.getSlots(date)
                .onSuccess { slots ->
                    _uiState.update { it.copy(slots = slots, isLoading = false) }
                }
                .onFailure { error ->
                    _uiState.update { it.copy(error = error.message, isLoading = false) }
                }
        }
    }
}

data class BookingUiState(
    val slots: List<TimeSlot> = emptyList(),
    val isLoading: Boolean = false,
    val error: String? = null
)

When MVVM Shines

  • CRUD-heavy screens — forms, lists, detail views
  • Simple state — a few independent pieces of data
  • Small teams — less ceremony, faster onboarding
  • Most screens in most apps — it covers 80% of use cases well

MVI: The State Machine

Model-View-Intent treats the UI as a state machine. Every user action is an Intent, every state transition is explicit, and the UI is always a pure function of state.

How I Structure MVI

class ClaimViewModel(
    private val repository: ClaimRepository
) : ViewModel() {

    private val _state = MutableStateFlow<ClaimState>(ClaimState.Initial)
    val state: StateFlow<ClaimState> = _state.asStateFlow()

    fun processIntent(intent: ClaimIntent) {
        when (intent) {
            is ClaimIntent.LoadPolicy -> loadPolicy(intent.policyId)
            is ClaimIntent.SelectClaimType -> selectType(intent.type)
            is ClaimIntent.SubmitDocuments -> submitDocuments(intent.docs)
            is ClaimIntent.ConfirmClaim -> confirmClaim()
        }
    }

    private fun confirmClaim() {
        val current = (_state.value as? ClaimState.ReadyToSubmit) ?: return
        _state.value = ClaimState.Submitting(current.claim)
        viewModelScope.launch {
            repository.submitClaim(current.claim)
                .onSuccess { result ->
                    _state.value = ClaimState.Submitted(result)
                }
                .onFailure { error ->
                    _state.value = ClaimState.SubmissionFailed(current.claim, error.message)
                }
        }
    }
}

sealed class ClaimState {
    object Initial : ClaimState()
    object Loading : ClaimState()
    data class PolicyLoaded(val policy: InsurancePolicy) : ClaimState()
    data class TypeSelected(val policy: InsurancePolicy, val type: ClaimType) : ClaimState()
    data class ReadyToSubmit(val claim: ClaimRequest) : ClaimState()
    data class Submitting(val claim: ClaimRequest) : ClaimState()
    data class Submitted(val result: ClaimResult) : ClaimState()
    data class SubmissionFailed(val claim: ClaimRequest, val error: String?) : ClaimState()
}

sealed class ClaimIntent {
    data class LoadPolicy(val policyId: String) : ClaimIntent()
    data class SelectClaimType(val type: ClaimType) : ClaimIntent()
    data class SubmitDocuments(val docs: List<Document>) : ClaimIntent()
    object ConfirmClaim : ClaimIntent()
}

When MVI Shines

  • Complex state transitions — insurance claim flows with conditional form steps, document uploads, and multi-stage validation
  • State needs to be predictable — every possible state is explicitly defined, critical for financial products where wrong state = wrong data submitted
  • Debugging is critical — you can log every intent and state change, essential when tracing why a claim submission failed or a policy loaded incorrectly
  • Multiple data sources — policy data from API, cached user details from local DB, and real-time claim status updates all merging into one screen

My Decision Framework

When a new feature or app lands on my desk, I ask three questions:

  1. How many distinct states can this screen be in? If I can count them on one hand, MVVM. If I need a diagram, MVI.
  2. Are there concurrent data sources? If the screen reacts to multiple streams of data, MVI handles the merging more gracefully.
  3. How important is state debugging? If a bug in this feature could cost users money or data, MVI's explicit state log is invaluable.

The Honest Truth

Most apps don't need MVI everywhere. I typically use MVVM as the default and switch to MVI for the screens that demand it. In the Kcuts Go app, 90% of screens are MVVM — only the booking confirmation flow with its payment integration uses MVI patterns.

The best architecture is the one your team can maintain at 2 AM when production is down. Choose clarity over cleverness.

Don't let architecture debates slow you down. Pick the simpler option, ship it, and refactor when complexity actually arrives — not when you imagine it might.
